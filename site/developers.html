API

API Reference

A RESTful API over HTTPS. JSON request and response bodies. Authentication via JWT bearer tokens or API keys. Rate limits vary by subscription tier.

Base URL

https://api.tminus.dev/v1

Authentication

JWT bearer tokens (short-lived) or API keys (long-lived, scoped)

Rate Limits

Free: 60 req/min

Pro: 600 req/min

Enterprise: Custom

Endpoints

Method Endpoint Description
POST /v1/auth/login Authenticate and receive JWT token pair
POST /v1/auth/refresh Refresh an expired access token
GET /v1/events List canonical events (filterable by date range, calendar, status)
POST /v1/events Create a new event with automatic policy projection
GET /v1/availability Query availability (mode=binary|probabilistic)
GET /v1/relationships List contact graph with drift scores and categories
POST /v1/relationships Add or update a relationship in the contact graph
GET /v1/drift-report Full drift report: fading relationships with severity
GET /v1/drift-alerts Active drift alerts requiring attention
POST /v1/commitments Create a time commitment (e.g., 20h/week for Client A)
GET /v1/commitments/:id/status Compliance status: compliant, under, or over threshold
POST /v1/scheduling/propose Propose meeting times with constraint satisfaction
POST /v1/scheduling/commit Commit a scheduling candidate to all calendars
GET /v1/constraints List active constraints (buffers, working hours, etc.)
POST /v1/constraints Create or update a scheduling constraint
MCP

MCP Tools

The Model Context Protocol is an open standard that lets AI assistants interact with external tools. T-Minus exposes 26 tools through its MCP server, giving any compatible AI agent full access to your temporal intelligence engine.

Point Claude, GPT, or any MCP-compatible assistant at https://mcp.tminus.dev and it can read your calendar, propose meeting times, track relationships, and manage scheduling constraints -- all through natural language.

Architecture

Zero servers. Pure edge.

T-Minus runs entirely on Cloudflare's global edge network. No origin servers, no containers, no orchestration. Compute and storage are colocated at the Durable Object level, giving each user a dedicated isolated runtime with zero network hop on the hot path.

Workers — Edge Compute

API OAuth Webhook Sync Consumer Write Consumer MCP Cron Push
↓ ↓ ↓

Durable Objects — Stateful Coordination

UserGraphDO (per-user) AccountDO (per-account) GroupScheduleDO (per-session)

Each DO instance has its own SQLite database -- 10 GB per user, colocated compute + storage

↓ ↓ ↓

Storage & Messaging

D1 (registry) R2 (proofs & audit) Queues (async processing) Workflows (multi-step)

Per-User Isolation

Each user gets a dedicated UserGraphDO instance with its own SQLite database. All per-user data -- events, policies, relationships, commitments -- lives in that single colocated store. No cross-tenant data leakage. No shared database bottlenecks.

→ 10 GB storage per user

→ Zero network hop for reads/writes

→ Natural tenant isolation

→ No cross-user contention

Event-Sourced Journal

Every mutation to the canonical event store produces an append-only journal entry recording the actor, change type, JSON patch, and reason. This powers audit trails, commitment compliance proofs, and temporal versioning.

{ "actor": "sync-consumer", "change_type": "UPDATE", "canonical_event_id": "evt_01HX...", "patch": { "start": "..." }, "reason": "provider_delta", "timestamp": "2026-02-15T..." }
Deployment

Self-Hosting

T-Minus is designed for Cloudflare deployment. A single wrangler deploy pushes everything to the edge. Environment configuration is straightforward.

Deployment Workflow

terminal
# Clone and install git clone https://github.com/tminus-dev/tminus.git cd tminus npm install # Configure environment cp .env.example .env.production # Edit .env.production with your secrets # Deploy to Cloudflare npx wrangler deploy --env production # Verify curl https://your-domain.com/v1/health -> { "status": "ok" }

Environment Configuration

Development

Local dev with wrangler dev. Uses local D1/R2/DO emulation. Hot reload for all workers.

Staging

Full Cloudflare deployment with staging credentials. Separate D1 databases, R2 buckets, and queue bindings from production.

Production

Production secrets via wrangler secret. Separate OAuth client IDs, encryption keys, and API configurations.
